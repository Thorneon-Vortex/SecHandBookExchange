#### 范式分析（PPT页面5）
**标题**：范式判定 - 以 Listing 表为例

**分析过程**：

> **Listing 表结构**：
> ```
> Listing(listing_id, seller_id, book_id, price, condition_desc, 
>         listing_type, status, post_time, description)
> ```
>
> **函数依赖集 F**：
> - listing_id → seller_id, book_id, price, condition_desc, listing_type, status, post_time, description
>
> **1NF 判定** ✅
> - 所有属性都是原子值（不可再分）
> - price 是 DECIMAL 类型，status 是 ENUM 类型，都是单值属性
>
> **2NF 判定** ✅
> - 主键是单属性 listing_id
> - 不存在部分函数依赖（单属性主键自动满足）
>
> **3NF 判定** ✅
> - 检查是否存在传递依赖：A → B → C
> - seller_id 不决定其他非主属性
> - book_id 不决定其他非主属性
> - 所有非主属性直接依赖于主键，不存在传递依赖
>
> **结论**：Listing 表满足第三范式（3NF）
>
> **其他表分析结果**：全部 7 张表均满足 3NF

### 第5部分：数据库高级特性（重点，3-4页）

#### 触发器（PPT页面12）
**标题**：数据库高级特性 - 触发器 (Trigger)

**什么是触发器**：
> 触发器是一种特殊的存储程序，当对表执行 INSERT、UPDATE、DELETE 操作时自动触发执行。
> 
> **优点**：
> - 自动化业务逻辑，减少应用层代码
> - 保证数据一致性，避免遗漏操作
> - 由数据库引擎执行，性能更高

**本项目使用的3个触发器**：

| 触发器名称 | 触发时机 | 功能描述 |
|-----------|---------|---------|
| after_order_insert | INSERT 后 | 创建订单后自动将书籍状态改为"已预定" |
| after_order_update_complete | UPDATE 后 | 订单完成后给买卖双方各增加5分信誉分 |
| after_order_update_cancel | UPDATE 后 | 取消订单后将书籍状态恢复为"在售" |

**触发器1代码示例**：
```sql
CREATE TRIGGER after_order_insert
AFTER INSERT ON `orders`
FOR EACH ROW
BEGIN
    -- NEW 关键字代表刚插入的数据
    UPDATE `listing`
    SET `status` = '已预定'
    WHERE listing_id = NEW.listing_id;
END$$
```

#### 存储过程（PPT页面13）
**标题**：数据库高级特性 - 存储过程 (Stored Procedure)

**什么是存储过程**：
> 存储过程是预编译的 SQL 语句集合，存储在数据库中，可以通过名称调用执行。
>
> **优点**：
> - 封装复杂业务逻辑，简化应用层代码
> - 支持事务控制，保证操作原子性
> - 预编译执行，减少网络传输，提高性能

**本项目使用的存储过程**：complete_transaction

**功能说明**：
| 项目 | 内容 |
|-----|------|
| 名称 | complete_transaction |
| 输入参数 | p_order_id (订单ID) |
| 输出参数 | p_result_code (0=成功, 1=状态错误, 2=数据库错误) |
| 输出参数 | p_result_message (结果描述) |

**存储过程代码**：
```sql
CREATE PROCEDURE complete_transaction(
    IN p_order_id INT,
    OUT p_result_code INT,
    OUT p_result_message VARCHAR(255)
)
BEGIN
    DECLARE current_status VARCHAR(50);
    DECLARE target_listing_id INT;
    
    -- 异常处理：回滚事务
    DECLARE EXIT HANDLER FOR SQLEXCEPTION
    BEGIN
        ROLLBACK;
        SET p_result_code = 2;
        SET p_result_message = '数据库错误';
    END;

    -- 查询订单状态
    SELECT order_status, listing_id 
    INTO current_status, target_listing_id
    FROM orders WHERE order_id = p_order_id;

    -- 状态校验
    IF current_status != '待确认' THEN
        SET p_result_code = 1;
        SET p_result_message = '订单状态不正确';
    ELSE
        -- 开启事务
        START TRANSACTION;
        
        -- 更新订单状态
        UPDATE orders SET order_status = '已完成' 
        WHERE order_id = p_order_id;
        
        -- 更新书籍状态
        UPDATE listing SET status = '已售出' 
        WHERE listing_id = target_listing_id;
        
        -- 提交事务（信誉分由触发器自动更新）
        COMMIT;
        SET p_result_code = 0;
        SET p_result_message = '交易成功完成！';
    END IF;
END$$
```

> 3. **自动化执行**：无需应用层手动调用信誉分更新逻辑
> 4. **可维护性**：业务规则变更只需修改数据库对象，无需改动代码

## 三、汇报演讲建议

### ⏰ 时间分配（假设15分钟）
| 部分 | 时间 | 备注 |
|-----|------|------|
| 背景介绍 | 1-2分钟 | 简洁明了 |
| 数据库设计 | 4-5分钟 | 重点讲解 |
| 功能演示 | 3-4分钟 | 可现场演示 |
| 高级特性 | 4-5分钟 | 触发器+存储过程 |
| 总结 | 1-2分钟 | 收尾 |

### 🎤 演讲技巧
1. **开场**：直接切入主题，不要过多寒暄
2. **重点**：突出数据库设计和高级特性（这是评分重点）
3. **演示**：如果允许，现场演示系统运行
4. **互动**：预留问答时间

### ❓ 可能的提问
1. "为什么选择使用触发器而不是在应用层处理？"
   - 答：减少应用层代码复杂度，保证数据一致性

2. "存储过程相比直接 SQL 有什么优势？"
   - 答：封装事务、减少网络传输、预编译提高性能

3. "如何保证并发情况下的数据一致性？"
   - 答：使用事务、行级锁、存储过程原子操作

4. "为什么选择 InnoDB 引擎？"
   - 答：支持事务、外键约束、行级锁、崩溃恢复

## 四、现场演示建议

如果允许现场演示，建议按以下流程：

### 演示流程（约3分钟）
1. **注册新用户**（30秒）
   - 填写学号、昵称、密码

2. **发布一本书**（45秒）
   - 选择分类、上传封面、填写价格

3. **搜索书籍**（30秒）
   - 关键词搜索、分类筛选

4. **下单购买**（30秒）
   - 查看触发器效果：书籍状态变为"已预定"

5. **确认交易**（45秒）
   - 查看存储过程效果：订单完成、书籍售出
   - 查看触发器效果：信誉分增加

### 演示前准备
- 提前插入一些测试数据
- 确保后端服务和数据库正常运行
- 准备 2 个测试账号（模拟买家和卖家）
